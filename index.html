ACMP / architecture narrative

从原始 stock vLLM，到 API Server多模态数据处理下沉，再到 mm-sidecar

ACMP 不是单点优化，而是一套“底座保底链 + 异步增强链”的完整多模态处理架构

这份页面不是摘要海报，而是一份面向新用户、设计评审和后续实现同学的完整讲解页。它按演进顺序回答： 原始 vLLM 多模态链路的问题是什么、现有 API Server多模态数据处理下沉 已经做对了什么、 为什么这些改进仍不足以彻底解决多并发图像 preprocess 波动，以及为什么需要引入 mm-sidecar 作为建立在现有底座之上的异步协同预处理增强层。

original vLLM pipeline API offload baseline async sidecar enhancement artifact + cache + fallback correctness first

这份文档的阅读目标

  • 让新同学从零看懂 ACMP 的完整心智模型。
  • 让设计评审能逐项检查边界、风险、fallback 和 correctness。
  • 让后续实现者能直接据此拆分模块与定义接口。

一句话结论

mm-sidecar 不是替代现有下沉方案的新主链，而是建立在 API Server多模态数据处理下沉 底座之上的异步预处理增强链； sidecar 能命中就吃收益，来不及就快速回退到底座保底链。

三段式演进

Stage 1 stock vLLM 将多模态热点压在 API Server。
Stage 2 API Server多模态数据处理下沉把主执行面移回 worker。
Stage 3 mm-sidecar 把 preprocess 提前做成异步平面与 CPU artifact 缓存层。
Guarantee sidecar 未命中或失败，立即退回 phase4/offload 保底链。

架构结论

Executive Summary

ACMP 的核心并不是“把 preprocess 搬到别处”，而是定义一个清晰的两层结构： 现有 API Server多模态数据处理下沉 继续作为稳定底座和 fallback 保底链， mm-sidecar 则作为异步协同预处理增强层，专门负责尽早启动 preprocess、提供 CPU artifact 级缓存复用、 统一治理多并发下的队列与生命周期，并在 TP Worker 到达消费点时按需汇合。

stock

原始链路

  • API Server 做读图、decode、resize、HF preprocess、hash、copy。
  • 多模态 CPU 热点集中在 API 侧。
  • 多并发下 TTFT 和 tail latency 波动明显。
api offload

现有底座

  • 把多模态主处理面下沉到 worker。
  • 为 http image 提供 local-ready 桥接能力。
  • 建立 stable metadata、identity 和 fallback 语义。
sidecar

异步增强层

  • 尽早启动 preprocess 并与主链重叠。
  • 提供 CPU NumPy artifact 缓存与统一治理平面。
  • TP Worker 最多短等 2ms，来不及就回 phase4/offload。
principle

首版原则

  • 边界清晰，只支持 local-ready image。
  • 结果必须与 stock / phase4 / sidecar 完全一致。
  • 收益增强失败时绝不能破坏原有稳定链。
01
背景与演进：为什么问题不是从 sidecar 才开始 先讲清楚原始链路和下沉底座，否则无法准确理解 sidecar 的定位。

在原始 stock vLLM 多模态链路中，API Server 不只做请求编排，它还承担了大量图像相关 CPU 热点： 图像读取、decode、resize、normalize、HF preprocess、multimodal hash 计算和部分 copy/serialize。 问题不是这些工作“本身不该做”，而是它们出现在了 API 侧关键路径上。 

Client Request
  |
  v
API Server
  |-- parse chat messages
  |-- read image / fetch image
  |-- decode image
  |-- resize / normalize
  |-- HF multimodal preprocess
  |-- compute multimodal hash
  |-- serialize / copy
  v
EngineCore / Scheduler
  v
TP Worker
  |-- consume already-preprocessed mm inputs
  v
ViT / LLM Forward

原始链路的问题画像

  • API Server 成为多模态 CPU 热点汇聚点。
  • 请求越重，TTFT 越容易直接被 API 侧 preprocess 拉长。
  • preprocess 与推理主链重叠不足，串行热点明显。
  • 多并发下，波动直接放大为 tail latency 问题。

API Server多模态数据处理下沉已经做对了什么

现有底座不是“又一条候选方案”，而是 ACMP 的前提。它已经把多模态主处理面下沉到 worker， 为 http image 提供了 local-ready 桥接能力，并建立了 local_path / media_uuid / source_key / request_media_index 这套稳定 metadata 语义。

为什么下沉方案还不够

即使热点已从 API 侧移到 worker，worker 侧 preprocess 仍然高度依赖图像尺寸、数量和分布而波动， 且 preprocess 启动时机仍偏晚，难以与 tokenizer、prompt build、engine submit 做更强重叠。

02
问题定义：这套方案到底要解决什么问题 不是泛泛谈“多模态慢”，而是明确列出系统级问题清单。

问题一：worker preprocess 波动大

图像尺寸、数量、纵横比与并发分布会直接放大到 ViT 前等待不均和 request tail latency。

问题二：preprocess 启动过晚

worker 拿到任务后才开始大规模 preprocess，导致主链与 preprocess 的重叠度仍然有限。

问题三：缺少独立治理平面

当前缺少专门面向 CPU preprocess artifact 的统一服务化治理层：队列、缓存、TTL/LRU、容量与路由都没有独立抽象。

问题四：需要 artifact 级复用

希望缓存的是 pixel_values、image_grid_thw、patch_grid 这类稳定产物， 而不是某个请求内临时过程状态。这样多轮对话和跨请求同源图像才能真正受益。

问题五：接入必须低风险

sidecar 的收益增强绝不能破坏现有链路。一旦未命中、超时、失败或整体不可用，请求必须快速回到 API Server多模态数据处理下沉 底座，必要时再回 stock。

03
方案目标与边界：哪些事情要做，哪些事情暂时不做 先把边界钉死，后面的架构与评测才不会发散。

目标

  • 在现有下沉方案兼容前提下引入图像异步 preprocess。
  • 让 API Server 主链与 preprocess 并行推进。
  • 提供 CPU artifact 缓存复用与统一治理。
  • 保留 TP / ViT-DP 既有消费语义不变。

MVP 输入边界

  • 仅支持 local-ready image。
  • image + local_path 可直接进入 sidecar。
  • image + http 需先经 phase4/offload materialize 为本地文件后再进入。

MVP 明确不做

  • 不支持 base64 image 进入 sidecar。
  • 不支持任何 video 进入 sidecar。
  • 不外置 ViT encode，不改写 EngineCore 主调度。

混合请求规则

只要请求中包含 base64 image 或任意 video，则整请求 bypass sidecar， 直接走 stock。首版不做一请求内媒体级混合拆分，以降低 correctness 与调试风险。

为什么边界如此保守

因为首版的关键不是“支持所有输入”，而是先证明 local-ready image 场景下，sidecar 可以在不破坏主链 correctness 的前提下带来真实收益。

04
总体架构：底座保底链与异步增强链如何协同 这是 ACMP 最核心的结构判断。

sidecar 不替代现有下沉方案，而是在现有保底链路之外，增加一个“尽早启动、异步执行、可缓存复用、TP Worker 按需汇合”的平面。 只要 sidecar ready，就吃增强收益；来不及时，马上回到底座保底链。 

Client Request
  |
  v
API Server
  |-- parse media / build metadata
  |-- for http image in current offload plan:
  |     materialize to local-ready file first
  |-- emit PREPARE to mm-sidecar as early as possible
  |-- tokenizer / prompt build / engine submit continue in parallel
  v
EngineCore / Scheduler
  |-- schedule as usual
  |-- carry lightweight metadata / handle only
  v
TP Worker
  |-- decide local shard ownership
  |-- GET_STATUS / FETCH_RESULT from mm-sidecar
  |-- ready -> use sidecar artifact -> ViT encode
  |-- not ready / timeout / sidecar unavailable
  |      -> fallback to API Server多模态数据处理下沉链
  |-- if unsupported there too -> stock
  v
LLM Forward / Decode

四条最重要的架构原则

  • 现有下沉方案是底座，不是 sidecar 的竞争者。
  • sidecar 是增强链，不是阻塞链。
  • metadata 必须保持与现有下沉链路兼容。
  • TP / ViT-DP 的分布式消费语义不能被 sidecar 改写。
05
端到端时序：请求进入后每一步到底怎么走 把时序写清楚，才能讨论 2ms wait budget 是否合理。
1
API Server 解析媒体项并建立基础 identity

构建 source_key、media_uuid、request_media_index，并复用现有 mm_hash 语义。

2
对 http image 先 materialize 成 local-ready 文件

sidecar 的输入契约是 local-ready image，不是原始 http transport。

3
API Server 拿到最小 metadata 后立刻发 PREPARE

PREPARE 采用 A 方案，绝不延后到更靠近 engine submit 才发。

4
API Server 主链继续推进

tokenizer / prompt build / engine submit 与 sidecar 后台 preprocess 并行发生。

5
sidecar worker 后台执行 preprocess

执行 read、decode、resize、HF preprocess，并持有 CPU NumPy artifact。

6
TP Worker 到达消费点后查询 sidecar 状态

先向 manager 走 GET_STATUS，仅当 ready 时再向 owner worker 发 FETCH_RESULT。

7
命中则直取 artifact，未 ready 则快速回退

TP Worker 只等 <= 2ms；若仍未 ready，就回到底座 phase4/offload 保底链，再必要时回 stock。

为什么 2ms 不矛盾

2ms 不是为了“现场把冷任务等出来”，而是为了接住 cache hit 或 near-ready 任务。真正让 sidecar 成立的关键不是等待更久， 而是 PREPARE 必须足够早地发出，从而让 preprocess 在主链前面提前开跑。

06
Artifact、Metadata 与 Identity：结果到底长什么样 sidecar 的核心交付物不是“某个对象”，而是稳定、可比较、可缓存的工件。

Artifact 格式选择

  • 对外标准格式固定为 CPU NumPy。
  • 不使用 GPU/NPU tensor 作为缓存协议格式。
  • 不以 Python pickle 作为主协议。

之所以选 NumPy，是因为它更中性、更适合作为跨进程 artifact 协议格式，也更容易明确约束 shape / dtype / layout。

Payload 最终目标

  • pixel_values
  • image_grid_thw
  • patch_grid

这三者是当前讨论中明确的目标产物，也是后续 correctness 对比的核心对象。

必须保留的底座兼容字段

local_path
现有下沉链路 fallback 的本地就绪对象语义。sidecar 不可用时，worker 直接利用它回到底座链。
media_uuid
请求内稳定媒体身份锚点，利于 trace 与排障，但不进入 cache key。
source_key
transport-aware identity，避免 local_path / http / base64 串扰。
request_media_index
请求内媒体顺序语义，与 TP / ViT-DP 消费规则保持对齐，但不进入 cache key。

sidecar 新增增强字段

mm_handle
运行时句柄，用于 GET_STATUS 与 FETCH_RESULT，不等于 cache key。
mm_schema_version
artifact 协议版本，用于消费前的格式兼容性校验。
preprocess_fingerprint
processor / resize / pad / normalize / patch / dtype / layout 的完整版本签名。
cache_key
source-level identity 与 preprocess 版本共同决定的缓存主键。

Artifact Header 建议字段

schema_version、processor_fingerprint、preprocess_fingerprint、 media_type、source_key、media_uuid、 original_size、resized_size、dtype、 shape、layout、created_at、expire_at。

07
cache key 与复用语义：跨请求怎么复用，为什么不追求最大化去重 这部分决定了 sidecar 缓存的边界是否稳定。 
cache_key = sha256(
    media_type
  + source_key
  + file_size
  + mtime_ns
  + preprocess_fingerprint
)

这个公式表达的设计判断

  • 复用语义是 source-level reuse，而不是内容级最大化 dedup。
  • file_size + mtime_ns 用于避免路径不变但文件已变化的误复用。
  • preprocess_fingerprint 用于隔离版本与配置漂移。

为什么不放入 media_uuid

media_uuid 更适合做请求内与 trace 身份锚点，放入 cache key 会破坏跨请求同源复用。

为什么不放入 request_media_index

request_media_index 是请求内顺序字段，放入 cache key 会让同一图像在不同请求顺序下无法复用。

复用能力的准确表述

sidecar 可以跨请求复用，但更准确地说是“同源引用级复用”，不是“同内容最大化复用”。对 MVP 来说这已经足够。

08
并发执行模型：多进程 sidecar 内部应该怎么跑 单并发只看链路，多并发才真正暴露方案优劣。 
mm-sidecar manager
  |- task registry
  |- cache metadata
  |- TTL / LRU / eviction
  |- owner routing
  |
  +-- worker-0 (compute + payload shard + ZMQ endpoint)
  +-- worker-1 (compute + payload shard + ZMQ endpoint)
  +-- worker-2 (compute + payload shard + ZMQ endpoint)
  +-- ...
  +-- worker-N (compute + payload shard + ZMQ endpoint)

最终并发模型

  • sidecar 采用固定多进程 worker pool，例如 32 进程占用 32 核。
  • 每个进程独立完成读图、decode、resize、HF preprocess、payload 持有与缓存写入。
  • manager 只维护 metadata、状态、TTL/LRU、capacity 与 owner routing。
  • payload 不回灌到 manager，直接由 owner worker 返回给 TP Worker。

为什么允许重复算

MVP 不做严格 inflight 去重。不同 worker 重复算同一媒体项是可接受的，但写缓存时若别的进程已发布成功，后来的结果不再重复发布。

为什么不是 manager 持 payload

如果 manager 同时承担 metadata、payload 写入和 payload 返回，会非常容易变成新的热点瓶颈。

为什么不做更复杂调度

像 small-first、等待态优先级提升、复杂 inflight dedup 都会显著增加复杂度，但并不是首版的关键收益来源。

09
传输协议：为什么不用 shm，为什么选 ZMQ + msgpack 协议与内存模型决定了 sidecar 的实现复杂度和可靠性上限。

为什么不走 shm payload

共享内存的主动驱逐、生命周期与容量治理复杂度很高，迟早会遇到“空间被挤爆”和残留对象问题，不适合做 MVP 主路径。

为什么选 ZMQ

vLLM 现有通信风格就有 ZMQ / msgpack 基础，sidecar 复用这一风格能降低接入和调试成本。

为什么是 REQ/REP

MVP 只需要支撑 PREPARE、GET_STATUS、FETCH_RESULT 三类同步请求，REQ/REP 语义清晰且足够简单。

最终协议风格

本地通信使用独立 ZMQ ipc://。结构化 header 采用 msgpack， 大 payload 采用 raw binary frames。

路由方式

TP Worker 先向 manager 查状态与 owner，再直连 owner worker 拉取 payload。 manager 不直接承担大数据出数。

10
等待、fallback 与失败处理：TP Worker 早到怎么办 这是 sidecar 成败最关键的极端场景。 
API Server gets minimal metadata
  |
  +-- emit PREPARE immediately
  |
  +-- tokenizer / engine submit continue in parallel

TP Worker receives mm_handle
  |
  +-- ready   -> consume sidecar result
  |
  +-- queued / running
  |      |
  |      +-- wait <= 2ms budget
  |             |
  |             +-- becomes ready -> consume
  |             +-- still not ready -> fallback phase4 / current offload path
  |
  +-- failed / expired / not_found -> fallback phase4 / current offload path
                           |
                           +-- if unsupported there too -> stock

明确结论

  • PREPARE 必须在 API Server 拿到最小 metadata 后立即发出。
  • TP Worker 总等待预算默认 2ms，并通过环境变量可配置。
  • 2ms 的意义是接住 cache hit 或 near-ready 任务，而不是等待冷任务完成。
  • fallback 后 sidecar 任务继续跑，完成后仍可写入缓存，为后续请求复用。

fallback 到哪一层

默认先 fallback 到 API Server多模态数据处理下沉 底座，再必要时回 stock。这正是 sidecar 低风险接入的关键。

为什么不取消 sidecar 任务

取消逻辑收益有限、复杂度高，而缓存住结果后，多轮对话和后续同源请求仍可能命中并摊平本次预处理代价。

11
缓存与生命周期：CPU artifact 池如何长期稳定工作 sidecar 的价值不只是“异步”，还包括“结果变成可治理资产”。

缓存准入规则

所有完成 artifact 都尝试进入缓存，不按文件大小做准入歧视。首版行为越简单，越利于解释、排障与回归。

逐出策略

统一采用 TTL + LRU。TTL 提供时间边界，LRU 提供空间压力下的热度淘汰能力。

为什么不做跨重启复用

cache 是内存态，sidecar 重启后变冷可接受。MVP 不为了持久化 cache 引入更复杂的一致性与版本问题。

总预算与 worker budget

统一配置例如 MM_SIDECAR_CACHE_GB=20、MM_SIDECAR_WORKERS=32。 MVP 默认均分到每个 worker：per_worker_budget = total / worker_count。

manager 需要看到的压力信息

每个 worker 至少上报 cache_bytes_used、artifact_count、 inflight_bytes 与 pressure_state，这样 eviction 决策才不会脱离真实压力。

删除中的借用保护

若 entry 正在被 FETCH_RESULT 读取，则 owner worker 标记为 borrowed，删除改为 pending_delete。

fetch 结束后再删

读取完成后立即删除并回传 delete-ack，避免 payload 返回过程中对象被删掉。

为什么不做全局 pin/unpin

复杂全局引用计数不是首版收益重点，而 borrowed / pending_delete 已足够覆盖 fetch 中途删除这一关键风险。

12
Prefix Caching、版本治理与 TP/ViT-DP 对齐 这部分决定 sidecar 是否真正能在现有系统里低风险落地。

Prefix Caching 的核心原则

sidecar 不重新发明多模态 hash。对于 extra_keys 依赖的多模态 hash，优先复用现有 API Server多模态数据处理下沉 / API Server 侧逻辑。

为什么仍然需要 preprocess_fingerprint

sidecar 缓存的不只是“这个输入是谁”，还包括“在某个 preprocess 版本和配置下产出的工件是什么”。 因此必须显式维护自己的 preprocess_fingerprint。

版本漂移如何隔离

将 schema_version、processor 版本、resize/pad/interpolation、normalize、patch size、dtype、layout 全部纳入 fingerprint。header 不匹配即视为 miss。

TP / ViT-DP 不被 sidecar 改写

sidecar 只输出全局稳定 preprocess 工件，不预先按 rank 切结果。TP Worker 仍按既有规则决定本 rank 消费哪些媒体项。

为什么这样最稳

它让 sidecar 与分布式执行语义尽可能解耦，避免首版把 cache、route 与 rank 语义绑死在一起。

13
可用性、监管与失败窗口：sidecar 挂了怎么办 既然 sidecar 是增强层，它就必须具备“随时失效但不拖垮主链”的能力。

sidecar 不可用时的行为

PREPARE 失败不阻塞 API Server 主链；GET_STATUS 或 FETCH_RESULT 失败则 TP Worker 快速 fallback 到 phase4/offload 底座。

短时失败窗口

API Server 与 TP Worker 都应实现轻量 breaker。建议环境变量：MM_SIDECAR_FAILURE_BACKOFF_MS=500。

外部监管方式

MVP 由 systemd 直接监管 mm-sidecar，负责自动拉起、异常重启、重启间隔和旧 IPC socket 清理。

为什么不在请求链里拉起 sidecar

请求链的职责是服务请求，不是做进程编排。把进程拉起与恢复逻辑放进主链，只会让 sidecar 从“增强层”变成“额外不稳定源”。 用 systemd 外部监管，职责最清晰、故障边界也最明确。

14
正确性验证：stock / phase4 / sidecar 必须完全一致 在 ACMP 里，correctness 不是附属任务，而是所有收益成立的前提。

artifact exact compare

必须比较 stock、API Server多模态数据处理下沉、mm-sidecar 三条链的 pixel_values、image_grid_thw、patch_grid、dtype、shape、layout。

ViT output exact compare

三条链进入 ViT 后的输出 tensor 必须逐项一致，不接受 near-exact、allclose 或“数值差不多”。

end-to-end fixed-case exact compare

固定输入、固定 prompt、固定推理配置后，最终输出必须 exact match。任何不一致都视为 correctness failure。

为什么一定要 exact

sidecar 是线上主链增强，不是研究 demo。只要 preprocess 产物或 ViT 输出有轻微漂移，就会进一步影响 prefix caching、 encoder cache、排障与回归判断，因此必须坚持 exact compare。

测试顺序建议

先在 local-ready image 场景做 artifact / ViT / end-to-end 三层对比， 先把 correctness 证清楚，再讨论性能收益和更宽输入边界。

15
性能收益分析：收益来自哪里，不来自哪里 只有把收益来源讲清楚，后续评测指标才不会选错。

可行性结论

该方案具备可行性，因为它建立在已有下沉底座之上，不需要推倒现有链路；它的增强点明确，fallback 语义清晰， 风险边界也容易控制。

收益主来源

收益来自更早启动 preprocess、更强的主链与 preprocess 重叠、更统一的 CPU artifact 治理与复用， 以及多并发下更可控的 preprocess 平面。

收益点一：PREPARE 提前发出

API Server 在 tokenizer / prompt build / engine submit 继续进行时就发 PREPARE，为 worker 抢出并行时间。

收益点二：artifact 级复用

多轮对话与跨请求同源图像可以直接复用 pixel_values / image_grid_thw / patch_grid，而无需重复 preprocess。

收益点三：独立治理平面

队列、cache、TTL/LRU、capacity、owner routing 被统一服务化治理，调优与观测都更清晰。

收益不来自哪里

不是因为 sidecar 改写了 EngineCore 调度器，不是因为外置了 ViT encode，也不是因为让 worker 长时间等待； 它的本质收益来自“同样正确的 preprocess 被更早启动，并被做成可复用的 artifact”。

什么时候收益可能不明显

极轻图像、低复用率、sidecar 命中率低且频繁 fallback 的场景中，收益可能不显著。因此评测必须同时看命中率、 near-ready 比例、fallback 比例和 cache 复用率。

使用环境变量 MM_SIDECAR_OBS_LEVEL=P0|P1|P2。P0 最小健康信号，P1 默认摘要级，P2 深度诊断。

先聚焦图像场景，先测 local-ready / local_path，先比较 stock / phase4 / sidecar 的 artifact、ViT 和 end-to-end exact match，再比较时延。

先不测什么

首版不要先把精力投入到 http 全链压测、video 和 base64，这些应该放到边界扩展阶段。

MVP 明确范围

本机 mm-sidecar service、local-ready image、CPU NumPy + metadata artifact、 ZMQ ipc:// + REQ/REP + msgpack + binary frames、TTL + LRU、systemd 监管、 以及严格 exact-match correctness。

后续再讨论的方向

包括 base64 image、video、更强 inflight dedup、跨机 sidecar、动态 budget 和更复杂调度增强。 这些都必须建立在图像 sidecar 主路径已稳定、收益已量化的前提之上。
